# Customer Journey — 고객 여정 4단계 키워드 가이드

`stage_configs.py` 기반. 고객이 구매 결정에 이르는 여정에서 각 단계별로 검색하는 키워드 패턴을 정의한다.

---

## 고객 여정 4단계 개요

| 단계 ID | 단계명 | 키워드 범주 | 고객 상태 |
|---|---|---|---|
| `info_search` | 정보탐색 | 정보탐색 | 문제를 어떻게 해결할지 찾는 중 |
| `solution_search` | 해결방안 탐색 | 정보탐색 | 여러 해결방법 중 무엇이 좋을지 비교 중 |
| `provider_search` | 업체/제품 검색 | 구매직전 | 방법 결정 후, 어디서 구매/등록할지 탐색 중 |
| `review_compare` | 후기 비교 | 구매직전 | 여러 업체 중 어디가 좋을지 후기 확인 중 |

> **참고**: 1단계(문제 인식)는 사용자 입력, 6단계(구매 결정)는 제외됨.

---

## BlogType별 단계 선택

```
INFO      → info_search, solution_search (정보탐색 키워드 사용)
PURCHASE  → provider_search, review_compare (구매직전 키워드 사용)
PORTFOLIO → provider_search, review_compare (구매직전 키워드 사용)
DAILY     → 키워드 단계 없음 (Step 2 생략)
```

### 선택 이유

- **INFO**: 정보를 찾고 있는 독자를 유입시켜 전문성을 보여주는 것이 목적 → 정보탐색 키워드
- **PURCHASE/PORTFOLIO**: 이미 구매/의뢰 의향이 있는 독자를 유입시키는 것이 목적 → 구매직전 키워드

---

## 단계별 상세

---

### 1. info_search — 정보탐색

**고객의 사고**: "이 문제를 어떻게 해결하지? 정보를 찾아보자"

**고객 상태**:
- 문제를 인식했지만 아직 해결방법을 모르는 단계
- 어떤 서비스/제품을 이용할지 전혀 결정하지 않은 상태
- 순수하게 정보를 원하는 상태

**키워드 특성**: `~방법`, `~하는법`, `~란`, `~효과`, `~원인` 등의 패턴

**키워드 예시** (수학 학원 업종 기준):
- "수학공부법", "수학독학", "수학잘하는법"

**키워드 생성 가이드**:
1. 독자가 어떤 문제로 검색할지 생각한다.
2. 문제 해결 방법을 묻는 형식으로 키워드를 만든다.
3. 업종/서비스명보다 **문제 자체** 중심으로 키워드를 구성한다.

**좋은 예** (헬스장):
- "뱃살빼는운동", "체지방줄이는법", "다이어트운동추천"

**나쁜 예** (구매직전 키워드가 섞인 경우):
- "헬스장추천", "헬스장가격" (이건 provider_search/review_compare)

---

### 2. solution_search — 해결방안 탐색

**고객의 사고**: "여러 해결방법 중에 뭐가 좋을까? 비교해보자"

**고객 상태**:
- 여러 해결방법이 있다는 것을 알고, 비교 중인 단계
- 아직 특정 업체/서비스를 고르지는 않은 상태
- 장단점을 비교하거나 어떤 방법이 자신에게 맞는지 탐색

**키워드 특성**: `~vs~`, `~비교`, `~장단점`, `~차이`, `~선택기준` 등의 패턴

**키워드 예시** (수학 학원 업종 기준):
- "수학내신올리기", "과외vs학원", "수학학원장단점"

**키워드 생성 가이드**:
1. 독자가 비교하려는 두 가지 옵션을 파악한다.
2. 비교, 장단점, 차이점을 묻는 형식으로 키워드를 만든다.
3. 독자가 어떤 기준으로 선택할지 고민하는 포인트를 키워드에 담는다.

---

### 3. provider_search — 업체/제품 검색

**고객의 사고**: "이 방법으로 결정했어. 어디서 구매/등록하지?"

**고객 상태**:
- 어떤 서비스/제품을 이용할지 결정한 상태
- 이제 구체적인 업체나 상품을 찾는 단계
- 높은 구매 의향을 가지고 있음

**키워드 특성**: `~추천`, `~업체`, `~전문`, `~잘하는곳` 등의 패턴

**키워드 예시** (수학 학원 업종 기준):
- "수학학원추천", "수학학원잘하는곳", "수학전문학원"

**키워드 생성 가이드**:
1. 독자가 업체/상품을 찾을 때 쓰는 표현을 생각한다.
2. 서비스명 + 추천/전문/잘하는곳 형식으로 키워드를 만든다.
3. 지역명이 있으면 "지역+서비스명" 형식도 고려한다.

**좋은 예** (헬스장):
- "헬스장추천", "헬스장가격", "강남헬스장", "PT가격"

**나쁜 예** (정보탐색 키워드가 섞인 경우):
- "뱃살빼는법", "다이어트방법" (이건 info_search)

---

### 4. review_compare — 후기 비교

**고객의 사고**: "여러 업체 중에 어디가 좋을까? 후기를 봐야겠다"

**고객 상태**:
- 여러 업체를 알고 있고, 어디가 더 나은지 비교하는 단계
- 실제 이용자 후기, 리뷰가 중요한 판단 기준이 됨
- 거의 결정 직전 상태

**키워드 특성**: `~후기`, `~리뷰`, `지역+업종`, `~경험담` 등의 패턴

**키워드 예시** (수학 학원 업종 기준):
- "강남수학학원", "수학학원후기", "수학강사추천"

**키워드 생성 가이드**:
1. 독자가 후기/리뷰를 검색할 때 쓰는 표현을 활용한다.
2. 지역명 + 업종명 조합, 또는 서비스명 + 후기/리뷰 형식으로 만든다.
3. 특정 업체/브랜드명을 포함하지 않는 일반적인 후기 키워드를 선택한다.

---

## 키워드 생성 실전 흐름 (본다 백엔드 4-노드 그대로)

본다 백엔드(`back/src/agents/keyword/graph.py`)의 LangGraph를 텍스트 워크플로우로 재현한다. **각 stage(`info_search`/`solution_search` 또는 `provider_search`/`review_compare`)별로 4개 노드를 순차 수행**한다.

### 🅐 generate_user_thought — 고객의 생각/행동 묘사

해당 stage에서 고객이 어떤 생각으로 검색을 시작하는지 2~4문장으로 묘사. 고객은 **아직 결론을 모르는 상태**여야 한다.

**입력**: stage_id, blog_type, my_service, 타입별 컨텍스트
- INFO → `user_provided_info`
- PURCHASE → `target_problem`
- PORTFOLIO → `portfolio_customer_problem` + `solution` + `result`

**규칙**:
1. 고객의 구체적 상황과 감정 포함
2. 실제로 검색창에 무엇을 입력할지 암시 ("~을 검색한다", "~를 알아본다")
3. 비교, 고민, 선택 과정 포함
4. 결론(해결책/사례)을 이미 아는 것처럼 묘사 금지

**Stage별 user_thought 예시**:

#### 정보탐색 (info_search)
- "수포자들을 위한 수학 공부법이 무엇인지 검색해보기 시작한다. 공부법에 동네 수학 다니기, 서울 유명 학원 다니기, 인강 듣기, 수학과외 방법이 있음을 확인한다."
- "주변 지인들은 무슨 운동을 하는지 살펴보다가 헬스, 복싱, 필라테스, 요가 효과 등을 비교한다."
- "처음에는 굳이 병원까지 갈 일인가 싶어 어금니 통증 원인이나 해결방법에 대해 검색한다."

#### 해결방안 탐색 (solution_search)
- "시간, 효과, 후기 등을 파악하여 오프라인 학원 수업과 온라인수업 비교를 한다. 결국, 여러 조건과 상황을 고려해 수학 인강 공부법을 선택한다."
- "복싱은 살을 빼기에 무난한지 검색한다. 헬스와 필라테스 가격 비교, 효과 비교 등을 알아본다."

#### 업체/제품 검색 (provider_search)
- "수포자 인강 추천, 고3 수학 인강 추천 등을 검색하며 어떤 수학인강을 들을지 알아본다."
- "필라테스에 관심이 생겨 집 주변 필라테스짐을 검색한다."

#### 후기 비교 (review_compare)
- "검색할 때마다 '이상한 강사'와 '리상한 강사'가 같이 노출된다. 둘의 경력과 후기/평을 살펴보며 비교한다."
- "고객들의 리뷰나, 평이 좋은 치과를 비교해본다."

### 🅑 convert_to_keywords — 검색 키워드 변환

user_thought를 네이버에서 검색량 높을 만한 키워드 5~7개로 변환.

**핵심 규칙**:
1. **2~3단어**로 구성 (공백 없이 붙여서) — 짧을수록 검색량 높음
2. 일반인이 검색창에 실제 입력할 만한 패턴
3. 검색량 높은 패턴 활용: "추천", "비용", "방법", "후기", "비교", "순위"
4. 5~7개 생성
5. `my_service` 도메인에 맞는 키워드만

**좋은 예** (AX컨설팅 기업):
- "AX컨설팅" (핵심어 + 패턴)
- "외주비용" (짧고 명확)
- "AI개발회사"
- "기업AI도입"

**나쁜 예**:
- "AI솔루션개발업체추천" — 4단어, 너무 김
- "맞춤형기업AI시스템구축" — 5단어, 아무도 이렇게 검색 안 함
- "프로그램" — 너무 일반적, 의미 없음

**핵심 원칙**:
- 서비스명이 길면 핵심 단어 1개만 추출
- 수식어(맞춤형/전문/고급) 금지

### 🅒 call_naver_api — 네이버 검색광고 API 호출

변환된 키워드 중 현재 인덱스의 키워드 1개를 hint로 `bonda:get_keywords` 호출.

```
입력: hint_keywords = ["키워드1"]  # 1개씩 순차 (최대 5개까지 가능)
출력: [
  {"keyword": str, "monthly_pc": int, "monthly_mobile": int, "total": int, "competition": str}
]
```

- 검색량 20 미만 자동 필터링됨
- total 기준 내림차순 정렬됨
- 결과 없으면 다음 검색 키워드(인덱스 +1)로

### 🅓 select_keywords — AI로 의미 있는 키워드 선별

naver API 결과 후보(최대 100개)에서 stage 카테고리(정보탐색/구매직전)에 부합하는 키워드만 선별.

**선택 규칙**:
1. **카테고리 목적에 맞는 키워드만** (가장 중요)
2. **`my_service`와 관련된 것만**
3. **비슷한 의미는 중복 선택 금지** (예: "AI기업" + "AI회사" → 1개만)
4. 유저의 검색 의도와 관련된 키워드 우선
5. 너무 일반적인 단일 용어 제외 ("다이어트", "운동" 등)
6. 경쟁 서비스/제품 키워드 제외
7. 적합한 키워드 없으면 빈 배열

**카테고리 설명**:
- **정보탐색**: 사용자가 아직 어떤 서비스/제품을 이용할지 결정 안 함, "~방법"·"~하는법"·"~효과"·"~비교" 같은 정보성 키워드
- **구매직전**: 사용자가 이용을 마음먹은 상태, "~추천"·"~후기"·"~가격"·"지역명+서비스명" 같은 구매 결정 키워드

**has_good_keywords 판단**:
- `true`: 검색 의도를 정확히 반영하는 키워드 3개 이상 / `my_service`를 직접 연상시키는 키워드 존재 / 카테고리에 완벽히 부합하는 키워드 3개 이상
- `false`: 너무 일반적 / 더 좋은 후보 가능성 / 선택된 게 1개 이하

**조기 종료 (MIN_GOOD_KEYWORDS = 3)**:
- 누적 좋은 키워드 ≥ 3개 AND `has_good_keywords=true` → 해당 stage 종료
- 부족하면 다음 검색 키워드(인덱스 +1)로 돌아가 🅒부터 반복
- 모든 검색 키워드 소진 시 종료

### 결과 통합

모든 stage의 결과 키워드를 합쳐 중복 제거 후 검색량 내림차순 정렬하여 사용자에게 표로 제시:

```
| 키워드 | 월 PC 검색량 | 월 모바일 검색량 | 합계 | 경쟁도 |
```

### 최종 키워드 선택 기준 (사용자 안내용)

1. **검색량**: 월 검색량이 적당히 있는 키워드 (너무 많으면 경쟁 심함)
2. **경쟁도**: "낮음" 또는 "중간"이 작성한 글이 상위 노출되기 유리
3. **관련성**: 실제 글 내용과 자연스럽게 연결되는 키워드
4. **의도 일치**: 선택한 stage(info_search 등)의 고객 의도와 일치

### MCP 미사용 시 폴백

`bonda:get_keywords` 호출이 실패하면 🅐 + 🅑 결과(검색량 없는 키워드)만 표로 제시. 검색량 컬럼은 `-`.
