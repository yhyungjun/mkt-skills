# P0 기획 질문 템플릿 (AskUserQuestion)

> 강의 사전예약·결제 랜딩 빌드를 시작할 때, **코드에 들어가기 전(Phase 0)** 사용자에게 확정받아야 할
> 질문 모음. 여기서 비우면 뒤(DB 컬럼·동선·결제·법무)가 전부 꼬인다. `AskUserQuestion` 툴로 묶어 묻되,
> 한 번에 너무 많이 묻지 말고 **묶음(아래 그룹) 단위**로 진행.
>
> 사용법: 각 질문의 "선택지 예시"는 보기로 제시하되 자유입력도 허용. 답을 `lib/config.ts` 상수 ·
> `.env` · 설문 컬럼 · 법무 초안으로 환원한다.

---

## 그룹 0 — 결제 방식 (★ 가장 먼저 · 전체 구성 분기)

> 이 답 하나가 페이지·lib·env·인증·트랙(A/B)을 통째로 결정한다 → **상품 정의보다 먼저 묻는다.**
> 상세 매핑은 `playbook.md`의 "STEP 0 — 결제 방식 선택" 표.

**0-1. 결제를 어떻게 받나 (택1)**
- 선택지:
  - **모델1 · 토스 PG 결제** — 사이트에서 즉시 카드결제. → Track A, `/pay`, 수강생 로그인·결제메일·웹훅 포함.
  - **모델2 · 계좌이체** — 신청 받고 계좌 입금확인. → Track B, `/apply`, 입금안내메일·Slack·어드민 confirmed. (토스/로그인/SMS 제외)
  - **모델3 · 외부 폼/링크** — 구글폼·외부 결제링크로. → 랜딩만, CTA가 외부 URL, P4~P10 대부분 생략.
  - **하이브리드** — 모델2로 먼저 오픈 → PG 통과 후 모델1로 승격.
- ⚠️ 고른 모델이 가리키는 모듈만 빌드(나머지 미구현). 모델1만 그룹 C(C1~C3)를 마저 답한다.

**0-2. 가격 변형 필요 여부 (결제 모델과 직교)**
- 무엇: 같은 랜딩을 가격만 다르게(공개가 vs 제휴/클럽 할인가) 여러 대상에 뿌릴지.
- 선택지: `단일 가격` / `변형 있음(공개 + 비공개 할인링크)` → 후자면 `code-templates/price-variant.md`.

## 그룹 A — 상품 정의

**A1. 강의명 / 기수 코드**
- 무엇: 화면 표기용 강의명, 내부 식별용 `cohort` 코드.
- 선택지 예시: `cohort` = `beta-2026` / `2026-1기` / `{slug}-2026`
- ⚠️ `cohort`는 **하드코딩 말고 설정값**으로(기수마다 코드 수정 방지).

**A2. 과정 형식**
- 무엇: 온라인/오프라인 제공 형태.
- 선택지 예시: `온라인만` / `오프라인만` / `온·오프라인 둘 다(선택)` / `온라인 라이브 + 녹화본`

**A3. 기간 / 구조**
- 선택지 예시: `4주 라이브` / `6주` / `1일 워크샵` / `8주(주1회)`

## 그룹 B — 가격 / 티어 / 인원

**B1. 정가 (형식별)**
- 무엇: 오프라인/온라인 각 정가.
- 선택지 예시: `오프라인 50만 / 온라인 50만` / `단일 가격` / `{입력}`

**B2. 얼리버드 할인 구조**
- 무엇: 티어별 할인율·인원.
- 선택지 예시: `tier1 20%(20명) → tier2 10%(20명) → 정가` / `얼리버드 단일 40%(N명 한정)` / `없음`
- ⚠️ 티어 인원 = `/api/counter`의 결제수로 산정. 인원·할인율을 `lib/config.ts`에.

**B3. 총 얼리버드 좌석 수**
- 무엇: 좌석 카운터 표시용 총 한정 수량.
- 선택지 예시: `40` / `30` / `50` / `{입력}`
- ⚠️ **진짜 한정 수량만**. 가짜 카운트는 신뢰·표시광고 리스크.

**B4. 보너스/번들 혜택 (선택)**
- 무엇: 금액 없이 얹는 추가 혜택.
- 선택지 예시: `제휴 툴 N개월 무료/할인` / `무료 자료·템플릿` / `없음`

## 그룹 C — 동선 (사전예약 vs 결제) — *모델1(PG)일 때만*

> 그룹 0에서 **모델1(토스 PG)**를 골랐을 때만 답한다. 모델2(계좌이체)·모델3(외부)는 이 그룹 생략.

**C1. 런칭 시점 동선**
- 무엇: `PAYMENT_ENABLED` 초기값 결정.
- 선택지 예시:
  - `사전예약 먼저 (PG 심사 대기 중)` → `PAYMENT_ENABLED=false`, CTA→`/signup`→`/preorder`
  - `결제 바로 오픈 (PG 이미 승인)` → `PAYMENT_ENABLED=true`, CTA→`/pay`
- ⚠️ PG 심사 안 끝났으면 무조건 사전예약 모드(가이드 `pg-review-guide.md`).

**C2. 결제 수단 / PG**
- 선택지 예시: `토스페이먼츠` / `기타 PG` / `미정(사전예약만 먼저)`

**C3. SMS 본인인증 사용 여부**
- 무엇: 온보딩에서 전화번호 인증할지.
- 선택지 예시: `OCTOMO 역방향 문자인증` / `전화번호만 수집(인증 없음)` / `미사용`

## 그룹 D — 설문 항목 (= DB 컬럼)

**D1. 사전예약 설문에서 수집할 필드**
- 무엇: 멀티선택. 고른 항목이 곧 `applications` 컬럼이 됨.
- 선택지 예시(복수 선택):
  - `과정 형태(온/오프)` `희망 요일` `희망 시간대` `지역` `직군/업종` `유입 채널`
  - `매출 규모` `기존 도구 사용 여부` `관심 강의` `자유 의견`
- ⚠️ **확정 목록을 마이그레이션에 명시**(코드 INSERT 동적생성 = 스키마 드리프트 함정).

**D2. 신청 폼 필수 항목**
- 선택지 예시: `이름·이메일·전화(기본)` / `+ 회사/소속` / `이메일만`

## 그룹 E — 수집 정보 & 동의 (PIPA)

**E1. 마케팅 수신 동의 받을지**
- 선택지 예시: `이메일 마케팅 동의(선택 항목)` / `이메일+SMS` / `안 받음`
- ⚠️ 마케팅 동의는 **필수와 분리(선택)** — 끼워넣으면 위법 소지.

**E2. 처리 위탁 외주 (개인정보처리방침에 기재)**
- 무엇: 실제 쓰는 외주를 빠짐없이.
- 선택지 예시: `PG(토스) / 메일(Resend·Google) / 호스팅(Cloudflare·Supabase) / 분석(GA)`

## 그룹 F — 도메인 & 발신

**F1. 배포 도메인**
- 선택지 예시: `example.ai` / `{입력}` / `미정(임시)`
- ⚠️ `NEXT_PUBLIC_SITE_URL`·`AUTH_URL`·OAuth redirect·메타데이터 base에 일관 적용.

**F2. 발신 이메일 주소**
- 선택지 예시: `noreply@{도메인}` / `hello@{도메인}` / `{입력}`
- ⚠️ Resend 도메인 인증(SPF/DKIM/DMARC) 필요 — 리드타임 있음.

**F3. 배포 타깃**
- 선택지 예시: `Cloudflare Workers` / `Vercel`
