# OCTOMO 설정 절차 (SMS 인증 · 선택)

> 휴대폰 본인확인용 SMS 인증. **선택 기능**(안 쓰면 생략 가능). env 키: `OCTOMO_API_KEY`.
> 이 스택은 **역방향 문자 인증** 방식을 쓴다(사용자가 지정 번호로 코드를 전송).
> 콘솔 UI 라벨은 자주 바뀌므로 "~ 찾기"로 기술.

## 1. API 키 발급
1. OCTOMO 서비스 가입 → 콘솔에서 **API 키 발급** 메뉴 찾기.
2. 발급된 키 → `OCTOMO_API_KEY` (서버 전용).

## 2. 역방향 문자 인증 흐름
일반적인 "코드 발송 → 입력"이 아니라 **사용자가 보내는** 방식:
1. 온보딩에서 사용자에게 **지정 수신번호 + 인증코드(4자리)**를 안내한다.
   - 코드는 `otp_codes` 테이블에 phone·code·`expires_at`(약 5분 TTL)로 저장.
2. 사용자가 자신의 휴대폰으로 그 번호에 코드를 문자 전송한다(발신번호 = 본인 번호).
3. 서버가 OCTOMO API(`checkOctomoMessage` 류)로 **해당 코드가 수신됐는지 폴링/확인**한다.
   - `POST /api/otp/verify` 가 OCTOMO를 조회 → 일치하면 통과.
4. 통과 시 `users.phone`, `users.phone_verified=true`, `users.onboarded_at` 갱신.
> 장점: 발신번호가 곧 본인 번호이므로 번호 소유를 강하게 검증.
> 흐름 가드: `otp_codes`의 `used`/`expires_at`으로 재사용·만료 처리.

## 3. 미사용 시 대안
`OCTOMO_API_KEY`를 비워두면 SMS 인증을 끄고 다음 중 하나로 대체:
- **단순 코드 발송/검증**: 다른 SMS 게이트웨이로 코드를 보내고 입력받아 비교(정방향).
- **전화번호만 수집**: 온보딩에서 phone만 받고 `phone_verified`는 false로 두기
  (본인확인이 필수 아닌 경우).
- env 미설정 시 OCTOMO 클라이언트가 stub로 동작하도록 두어 빌드가 깨지지 않게 한다.

## 흔한 함정 요약
- 코드 TTL(약 5분) 만료 후 검증 시도 → 실패. UI에 재발급/재시도 제공.
- 같은 코드 재사용 → `used` 플래그로 차단.
- 키 미설정인데 인증을 필수로 막아둠 → 가입 불가. 미사용이면 대안 흐름으로 우회.
- OCTOMO 조회는 비동기 수신 기반이라 즉시 안 잡힐 수 있음 → 짧은 폴링/재시도 UX.
